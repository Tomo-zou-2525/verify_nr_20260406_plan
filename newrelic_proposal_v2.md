# 第一章：あるべき運用の姿

## 1-1. 私たちが目指す運用とは何か

現状の運用は「要件を満たす」という意味において、間違っていません。  
しかし、市場・顧客の期待値は確実に上がっていきます。

> **「これから価値ある運用部隊として進化するために、今、可視性の基盤を作る必要がある。」**

私たちが目指すべき運用の姿を、以下の5つの要素で定義します。

| 要素 | 定義 | 現状との距離 |
|---|---|---|
| ① 再現性 | 誰がやっても同じ品質で対応できる | △ |
| ② 可視性 | システムの状態・過去の経緯が誰でも見える | ✕（最も遠い） |
| ③ 学習性 | インシデントや知見が組織の資産として蓄積される | △ |
| ④ 持続性 | 特定の人に依存せず、チームとして継続できる | △ |
| ⑤ 主体性 | 現場メンバーが根拠を持って気づきを発信できる | ✕ |

### 5つの要素の連鎖

```
⑤ 主体性　が起点
　↓ 主体性があると
② 可視性　現場の気づきが組織に届く
　↓ 可視性があると
③ 学習性　気づきがナレッジとして蓄積される
　↓ 学習性があると
① 再現性　誰でも同じ品質で対応できる
　↓ 再現性があると
④ 持続性　人に依存しない組織運用が実現する
```

> **主体性が起点で、持続性が終点。**
> この連鎖を機能させることが、価値ある運用部隊への進化です。

## 1-2. 現状の組織課題：3つのレイヤー

あるべき姿に対して、現状の課題を3つのレイヤーで整理します。

### Layer A：開発チームの課題

| 現状 | 課題 |
|---|---|
| 要件を満たすことが主目的であり、それは現時点では正しい判断 | 市場・顧客の期待値上昇に対応する準備ができていない |
| パフォーマンス改善・継続的な品質向上は要件外のことが多い | 改善のための計測・観測がそもそも設計されていない |
| 要件定義の段階で運用品質が議題に上がらない | 運用性・可観測性がシステム設計に組み込まれない |

### Layer B：運用部隊の課題

| 現状 | 課題 |
|---|---|
| CloudWatchのみで案件ごとに監視が分断 | 複数案件を横断した可視性がなく、対応が後手に回る |
| ナレッジが特定メンバーの頭の中に閉じている | 担当者離脱で知見が失われ、同じトラブルが繰り返される |
| 派遣・契約メンバーが実務の中心を担う | 指示されたタスクのみをこなす構造になり、主体性が生まれにくい |
| 気づきの報告がやる気のある人頼み | 計測できないから改善できない。見えないから提案できない |

### Layer C：横断的・組織的な課題

| 現状 | 課題 |
|---|---|
| 開発と運用の間にフィードバックループが存在しない | 運用知見が開発品質に還元されず、組織全体の成長につながらない |
| ナレッジを書く文化・インセンティブがない | 書く人だけが損をする構造になっており、組織学習が起きない |
| 人員ローテーションがなく案件内にナレッジが閉じる | 横展開ができず、別案件で同じ失敗が繰り返される |

---

## 1-3. オブザーバビリティの位置づけ

上記3層の課題に対して、オブザーバビリティ（可観測性）は以下のように機能します。

```
オブザーバビリティが解決すること
　├─ 可視性：システムの状態をデータとして誰でも見られる状態にする
　├─ 主体性：現場メンバーが「感覚」ではなく「データ」で報告・提案できる
　└─ 学習性：インシデントの文脈・原因・対処がデータとして残る

オブザーバビリティだけでは解決しないこと
　├─ ナレッジを書く文化・インセンティブの設計
　├─ 担当者アサインの標準化
　└─ 開発・運用間のフィードバックプロセスの設計
```

> **オブザーバビリティは「可視性」を起点に5つの連鎖を動かすための基盤インフラです。**
> 「ツールを入れたい」ではなく、**組織の構造的課題に対処する最初の一手**として位置づけます。

# 第二章：現状の監視体制と隠れたコスト

## 2-1. 現状の監視体制

| 項目 | 現状 |
|---|---|
| 主要監視ツール | CloudWatch（クラウドネイティブ監視のみ） |
| 監視の範囲 | 案件ごとに分断。横断的な可視性なし |
| 障害原因特定 | 手動でログを追う。1時間以上かかるケースが常態化 |
| ナレッジ管理 | Jira / Confluence を一部利用。書く文化がなく形骸化 |
| 再発防止 | 特定メンバーの記憶・経験に依存（属人化） |

## 2-2. 隠れたコスト

### 工数コスト試算（概算）

| 項目 | 現状 | 月間工数試算 |
|---|---|---|
| 障害対応（検知〜復旧） | 月間XX件 × 平均XX時間 × XX名 | 約XXX時間/月 |

| 深夜・休日の緊急対応 | 月間XX件 | 約XXX時間/月 |
| **合計** | | **約XXX時間/月** |

> 💡 エンジニア平均単価をXX円/時間とすると、月間**約XXX万円相当**の工数が監視・障害対応に消費されています。

### 数字に表れにくいリスク

- 特定メンバーへの集中による**バーンアウトリスク**
- 担当者不在時の**対応空白リスク**

> ※ 上記工数データは現在集計中です。本格評価フェーズにて正確な数値を提示します。

# 第五章：New Relicで変わること

## 5-1. Before / After 比較

| | Before（現状） | After（New Relic導入後） |
|---|---|---|
| **障害検知** | CloudWatchアラート、案件ごとに分断 | 全案件を横断した統合監視・異常の自動検知 |
| **原因特定** | ログを手動で追う、1時間以上 | トレース・ログ・メトリクスを一画面で相関分析 |
| **再発防止** | 担当者の記憶・属人的なナレッジ | ダッシュボードで傾向を可視化、チームで共有 |
| **開発へのFB** | 遅い・少ない・見えない | デプロイとパフォーマンスを紐付けて即フィードバック |
| **夜間・休日対応** | 特定メンバーに集中 | アラート設計により初動を自動化・分散化 |
| **現場の主体性** | 感覚・善意による報告 | データを根拠とした提案・改善サイクルへ |

## 5-2. あるべき運用の姿との対応

| あるべき姿の要素 | New Relicの貢献 |
|---|---|
| ② 可視性 | ◎ 直接解決。全案件の状態をリアルタイムで可視化 |
| ⑤ 主体性 | ○ データを持つことで、根拠ある発信が可能になる |
| ③ 学習性 | ○ インシデントの文脈・原因・対処がデータとして残る |
| ① 再現性 | △ 可視性・学習性の向上を通じて間接的に貢献 |
| ④ 持続性 | △ 属人化の解消を通じて間接的に貢献 |
